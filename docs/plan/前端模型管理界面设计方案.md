# 前端模型管理界面设计方案

## 一、 接口深度解析 (`@/ai_werewolf_core/api/routes/models.py`)

当前后端提供了以下三个核心 RESTful API 接口用于模型配置管理：

1.  **获取模型列表 (`GET /api/models`)**
    *   **响应**: `List[ModelConfigResponse]`
    *   **字段**: `id`, `provider`, `name`, `base_url`, `model_name`, `temperature`, `max_tokens`, `timeout`
    *   **注意**: 响应中**不包含** `api_key`，这是后端的安全设计，前端在展示列表时无需也无法显示真实的 API Key。

2.  **创建/新增模型 (`POST /api/models`)**
    *   **请求体**: `ModelConfigCreate` (包含 `id`, `provider`, `name`, `api_key`, `base_url`, `model_name`, `temperature`, `max_tokens`, `timeout`)
    *   **行为**: 后端会检查 `id` 是否重复，并调用 `encrypt_api_key` 对 `api_key` 进行加密后存入数据库，最后触发 `ModelRegistry.reload()` 刷新内存中的模型注册表。
    *   **响应**: `ModelConfigResponse` (不含 `api_key`)

3.  **删除模型 (`DELETE /api/models/{model_id}`)**
    *   **行为**: 根据 `model_id` 删除数据库记录，并触发 `ModelRegistry.reload()`。
    *   **响应**: `{"status": "success"}`

*注：当前后端暂未提供 `PUT /api/models/{model_id}` (编辑) 和专门的连通性测试接口。前端的“编辑”功能可通过“删除旧配置 + 创建新配置”的组合操作实现（或建议后端后续补充 PUT 接口）；连通性测试可建议后端增加一个 `/api/models/test` 接口，前端先按此规划。*

## 二、 整体视图布局与组件规划

模型管理控制台（Model Management Console）建议采用 **左右分栏** 或 **主列表+抽屉/弹窗** 的布局模式。考虑到配置项较多，推荐使用 **主列表 + 侧边抽屉 (Drawer)** 的交互形式。

### 1. 核心组件划分

*   **`ModelManagementView.vue` (主视图)**
    *   负责整体页面布局、状态管理（Pinia Store 接入）和全局错误拦截。
*   **`ModelList.vue` (模型列表组件)**
    *   **展示形式**: 卡片网格 (Grid of Cards) 或 数据表格 (Data Table)。推荐使用卡片，更符合“模型实例”的具象化认知。
    *   **卡片内容**:
        *   模型名称 (`name`) & 供应商 Logo/标签 (`provider`)
        *   模型 ID (`id`)
        *   基础配置摘要 (`model_name`, `temperature`)
        *   操作区：[测试连接]、[编辑]、[删除]
*   **`ModelConfigForm.vue` (配置表单组件 - 抽屉内)**
    *   用于新增或编辑模型配置。
    *   **表单项**:
        *   **基础信息**: 模型 ID (唯一标识，编辑时禁用)、显示名称、供应商 (下拉单选: OpenAI, Anthropic, Local 等)。
        *   **连接配置**: Base URL、API Key (密码输入框)。
        *   **推理参数**: Model Name (如 gpt-4-turbo)、Temperature (滑块 0-2)、Max Tokens (数字输入)、Timeout (数字输入)。
    *   **操作区**: [取消]、[保存并测试]、[仅保存]。

## 三、 敏感信息安全与脱敏显示

### 1. API Key 的输入与展示
*   **输入态**: 在 `ModelConfigForm` 中，API Key 必须使用 `<input type="password">`，并提供一个“眼睛”图标用于临时切换可见性（仅在输入时）。
*   **编辑态**: 由于后端 `GET` 接口不返回 `api_key`，当用户点击“编辑”打开表单时，API Key 字段应显示为**占位符**（如 `••••••••••••••••`）或留空，并提示：“出于安全原因，API Key 不予显示。若需修改，请重新输入；若留空，则保持原 Key 不变。”（*注：这需要后端 PUT 接口支持部分更新，若当前仅有 POST，则编辑时必须要求用户重新输入，或前端提示“当前版本编辑需重新输入 API Key”*）。
*   **传输态**: 强制要求前端部署在 HTTPS 环境下，确保 API Key 在传输过程中通过 TLS 加密。后端接收后已通过 `crypto.py` 的 Fernet 进行对称加密落库。

## 四、 连通性测试与异常交互机制

### 1. 连通性测试 (Connectivity Test)
*   **交互入口**: 列表卡片上的 [测试] 按钮，或表单底部的 [保存并测试] 按钮。
*   **UI 反馈**:
    *   **加载中 (Loading)**: 点击后按钮变为 Spinner 状态，卡片/表单局部增加半透明遮罩，防止重复点击。
    *   **成功 (Success)**: 弹出绿色 Toast/Message 提示“连接成功，模型响应延迟: xxx ms”。
    *   **失败 (Error)**: 弹出红色 Alert，展示具体的错误原因（如“API Key 无效”、“网络超时”、“Base URL 无法解析”）。
*   **重试机制**: 失败提示框中提供 [重试] 按钮。

### 2. 全局错误拦截与处理
*   **Axios 拦截器**: 在 `@/frontend/src/api/client.ts` 中配置全局响应拦截器。
*   **状态码处理**:
    *   `400 Bad Request`: 捕获如 "Model ID already exists" 的业务错误，直接映射到表单字段的错误提示上。
    *   `401/403`: 提示认证失败。
    *   `500+`: 提示“服务器内部错误，请检查后端日志”。
    *   `Timeout`: 提示“请求超时，请检查网络或模型 Base URL 设置”。

## 五、 数据流转与状态管理 (Data Flow)

1.  **状态定义 (Pinia Store: `useModelStore`)**
    *   `models`: 存放 `ModelConfigResponse` 数组。
    *   `isLoading`: 列表加载状态。
    *   `isSaving`: 表单提交状态。
    *   `testStatus`: 记录各模型的测试连通性状态（`Record<model_id, 'success' | 'error' | 'testing' | 'idle'>`）。

2.  **流转流程 (新增模型为例)**
    *   **Step 1**: 用户在 `ModelConfigForm` 填写数据，前端进行本地校验（必填项、数值范围）。
    *   **Step 2**: 点击 [保存]，触发 Store 的 `createModel(data)` Action。
    *   **Step 3**: Axios 发送 `POST /api/models` 请求。
    *   **Step 4**: 后端校验 ID，加密 API Key，落库，重载 Registry，返回脱敏后的配置。
    *   **Step 5**: 前端 Axios 拦截器放行，Store 将新模型 push 到 `models` 列表中，关闭抽屉，弹出成功提示。

3.  **流转流程 (编辑模型 - 变通方案)**
    *   由于当前无 `PUT` 接口，前端的“编辑”逻辑可设计为：
        *   打开表单，填充旧数据（API Key 留空）。
        *   用户修改后点击保存。
        *   前端先调用 `DELETE /api/models/{old_id}`。
        *   成功后再调用 `POST /api/models` 写入新数据。
        *   *(强烈建议后续在后端补充 `PUT /api/models/{model_id}` 接口以实现真正的平滑更新)*。
