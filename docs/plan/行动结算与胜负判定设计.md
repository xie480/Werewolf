# 行动结算与胜负判定设计 (Action Resolver & Win Evaluator)

## 1. 模块职责概述
本模块是 Game Engine 中最复杂的逻辑处理中心。
- **Action Resolver (行动解析与结算器)**：负责接收 Agent 提交的意图（Intent）。在夜晚串行执行的架构下，它作为"夜间状态暂存器 (Draft State Manager)"，维护预期死亡名单，并在夜晚结束时统一结算。
- **Win Condition Evaluator (胜负判定器)**：在每个关键阶段结束时（如天亮后、投票出局后），独立评估当前存活玩家状态，判断是否触发游戏结束条件（屠边或屠城）。

## 2. 核心架构设计

### 2.1 Action Resolver

**文件路径**: `ai_werewolf_core/core/engine/resolver.py`

- **串行状态暂存**：夜晚阶段严格串行执行（狼人 → 女巫 → 预言家）。
- **状态快照与应用**：结算过程中，先在内存中维护一个"预期死亡名单"（`pending_deaths`）。狼人刀人加入名单，女巫救人移出名单，女巫毒人加入名单。待所有夜间动作结算完毕后，统一应用状态变更并生成天亮信息。
- **PlayerStatusManager 集成**：死亡结算时，调用 `role.die()` 标记内存状态后，通过 `PlayerStatusManager.mark_dead()` 同步更新 Redis BitMap (`werewolf:alive:{game_id}`)，确保多 Worker 进程获取到一致的存活状态。
- **夜晚周期管理**：
  - `begin_night()`：每轮夜晚开始前调用，清空 `pending_deaths`、`pending_actions`、`_current_night_wolf_target` 等草稿状态。
  - `submit_action()`：阶段校验 + 角色校验（`role.validate_action()`）+ 目标校验 → 暂存动作 + 更新草稿死亡名单。
  - `resolve_night_actions()`：按"狼刀 → 女巫救 → 女巫毒"规则计算最终死亡名单，执行 `die()` + `PlayerStatusManager.mark_dead()` + 发布 `PLAYER_DEATH` 事件。

**死亡结算的跨模块协作**:
1. `ActionResolver` 调用 `target_role.die()` 标记内存状态
2. `ActionResolver` 调用 `PlayerStatusManager.mark_dead(game_id, player_id, seat_number)` 同步 Redis BitMap（SETBIT 0）
3. `PlayerStatusManager` 异步更新 DB `PlayerRecord.is_alive=False`（最终一致性）
4. `ActionResolver` 通过 EventBus 发布 `PLAYER_DEATH` 事件

### 2.2 Win Condition Evaluator

**文件路径**: `ai_werewolf_core/core/engine/evaluator.py`

- **独立无状态**：判定器不保存游戏状态，所有方法均为静态方法。每次调用时传入当前的存活玩家列表和角色信息，确保纯函数式、可测试、无副作用。
- **存活状态读取**：`WinEvaluator` 直接读取 `role.is_alive`（进程内存中的快速路径）。跨 Worker 的一致性由各结算模块在死亡时同步 Redis BitMap 来保证——同一 Worker 内的角色实例 `is_alive` 属性与 Redis BitMap 保持同步。
- **神职集合（GOD_ROLES）**：使用 `frozenset` 定义神职角色集合（`SEER`, `WITCH`, `HUNTER`），不可变且支持 O(1) 查找。
- **屠边规则**：判断好人阵营（神职全死或平民全死）或狼人阵营（狼人全死）是否满足失败条件。
- **屠城规则（可选）**：当狼人存活数量 >= 好人存活数量时，狼人直接获胜。
- **判定优先级**：
  1. 狼人全死 → 好人胜（优先级最高）
  2. 神职全死 或 平民全死 → 狼人胜（屠边）
  3. (屠城模式) 狼人数量 >= 好人数量 → 狼人胜
  4. 否则 → 游戏继续

## 3. 接口与方法定义

### 3.1 ActionResolver 核心接口

```python
class NightResolveResult:
    """夜晚结算结果 —— 封装最终死亡名单和死亡原因。"""
    final_deaths: List[str]         # 最终死亡玩家 ID 列表
    death_reasons: Dict[str, ActionType]  # player_id → 死亡原因
    saved_players: Set[str]         # 被女巫解药救活的玩家
    poisoned_players: Set[str]      # 被女巫毒药毒杀的玩家
    wolf_target: Optional[str]      # 狼人原始刀人目标（可能被救活）
    is_peaceful_night: bool         # 是否为平安夜

class ActionResolver:
    def __init__(self, game_id: str, event_bus: EventBus):
        self.pending_deaths: Dict[str, ActionType] = {}   # 预期死亡名单草稿
        self.pending_actions: List[AgentAction] = []      # 已收集的动作列表
        self._current_night_wolf_target: Optional[str] = None  # 狼人原始刀人目标
        self._player_status = PlayerStatusManager()       # Redis BitMap 同步器

    def begin_night(self) -> None:
        """清空上一轮的草稿状态，开启新的夜晚结算周期。"""

    def submit_action(self, action, roles, current_phase) -> bool:
        """
        接收并暂存 Agent 提交的动作。
        校验顺序：阶段匹配 → 行动者存活 → Role System 合法性 → 目标有效性
        校验通过后加入 pending_actions，并更新 pending_deaths 草稿。
        """

    async def resolve_night_actions(self, roles) -> NightResolveResult:
        """
        夜晚统一结算：按狼刀→女巫救→女巫毒规则计算最终死亡名单。
        对每个死亡目标：调用 die() → PlayerStatusManager.mark_dead() → 发布 PLAYER_DEATH
        """
```

## 4. 关键数据流转过程

### 4.1 夜晚行动收集与暂存
1. 状态机依次进入 `NIGHT_WOLF_ACT` → `NIGHT_WITCH_ACT` → `NIGHT_SEER_ACT` 阶段。
2. Agent 提交动作，API 层调用 `ActionResolver.submit_action()`。
3. Resolver 校验合法性（阶段 + 角色 + 目标），暂存到 `pending_actions`。
4. Resolver 根据动作类型更新 `pending_deaths` 草稿：
   - `WOLF_KILL`：目标加入 `pending_deaths`，记录 `_current_night_wolf_target`
   - `WITCH_SAVE`：从 `pending_deaths` 中移除狼人目标（仅当目标匹配）
   - `WITCH_POISON`：目标加入 `pending_deaths`
5. `SEER_CHECK` 不修改 `pending_deaths`，仅记录以供后续生成验人信息。

### 4.2 夜晚统一结算
1. 状态机进入 `NIGHT_RESOLVE` 阶段，调用 `resolve_night_actions()`。
2. Resolver 读取 `pending_deaths` 作为最终死亡名单。
3. 对每个死亡目标：
   - 调用 `role.die()` 标记内存 `is_alive=False`
   - 调用 `PlayerStatusManager.mark_dead()` 同步 Redis BitMap（SETBIT 0）
   - 异步更新 DB `PlayerRecord`（最终一致性）
4. 发布 `PLAYER_DEATH` 事件（`visibility=PUBLIC`），payload 包含死亡原因。
5. 发布 `NIGHT_RESOLVE_EVENT` 事件（向所有玩家广播天亮信息）。

### 4.3 胜负判定
1. 结算完毕后，引擎调用 `WinEvaluator.evaluate(roles)`。
2. Evaluator 遍历 `roles`，统计存活狼人/神职/平民数量（读取 `role.is_alive`）。
3. 应用屠边规则（或屠城规则）判定胜负。
4. 如果返回非 `None`，引擎调用 `LifecycleManager.end_game(winner)`，状态机流转至 `SETTLING → FINISHED`。
5. 否则，状态机流转至 `DAY_START`（进入白天讨论）。

## 5. 依赖关系说明
- **Role System (`ai_werewolf_core/core/engine/roles/`)**：强依赖。Resolver 调用 `role.validate_action()` 校验动作合法性；Evaluator 读取 `role.is_alive`、`role.role_type`、`role.faction` 进行统计。
- **PlayerStatusManager (`ai_werewolf_core/utils/player_status.py`)**：强依赖。死亡结算时通过此模块同步 Redis BitMap 存活状态，确保多 Worker 一致性。
- **Event Bus (`ai_werewolf_core/core/event/bus.py`)**：强依赖。结算结果（`PLAYER_DEATH`、`NIGHT_RESOLVE_EVENT`）通过事件总线广播给 Agent 和前端。
- **DB Models (`ai_werewolf_core/db/models.py`)**：间接依赖。PlayerStatusManager 异步更新 `PlayerRecord.is_alive`（最终一致性，不阻塞游戏流程）。
