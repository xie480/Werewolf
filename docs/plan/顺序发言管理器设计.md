# 顺序发言管理器设计

## 1. 架构定位与设计目标

在狼人杀对局中，白天的讨论阶段（`DAY_DISCUSSION`、`DAY_PK_DISCUSSION`、`LAST_WORDS`）要求玩家按照特定的顺序（通常是座位号升序）依次发言。
之前的实现中，系统在进入发言阶段时，会并发地为所有存活玩家分发决策任务，导致所有玩家同时思考并直接提交发言，这破坏了游戏规则。

为了修复此问题并遵循单一职责原则 (SRP)，我们引入了 **发言管理器 (`SpeechManager`)**。

**核心职责**:
1. **初始化发言队列**：在进入发言阶段时，按座位号升序构建玩家发言顺序队列。
2. **轮流发言控制**：每次仅允许当前队列头部的玩家发言，其余玩家等待。
3. **发言完成后推进**：当前玩家发言完毕后，自动弹出队列头部并触发下一个玩家。
4. **提前结束检测**：当队列为空（所有玩家已发言完毕）时，满足提前结束条件，推进到下一阶段。

## 2. 核心组件设计

### 2.1 发言管理器 (SpeechManager)

**文件路径**: `ai_werewolf_core/core/engine/speech_manager.py`

**Redis List 无状态存储设计**:

`SpeechManager` 不在实例变量中保存队列状态，所有队列数据存储于 Redis List：
- **Key**: `werewolf:speech_queue:{game_id}`
- **内容**: 玩家 ID 字符串列表，按发言顺序排列（左端为即将发言的玩家）
- **TTL**: 24 小时（86400 秒）

**Why 用 Redis List**:
1. `LPOP`/`RPUSH` 是原子操作，天然适合轮序控制。
2. 支持多 Worker 场景，不会出现重复弹出或跳过玩家。
3. 提供 `LLEN` 快速查询剩余待发言人数。

**核心接口设计**:
```python
class SpeechManager:
    def __init__(self, game_id: str, event_bus: EventBus):
        self.game_id = game_id
        self.event_bus = event_bus
        self._redis = None

    async def init_queue(self, roles: Dict[str, BaseRole], current_round: int, phase: GamePhase) -> None:
        """初始化发言队列：按座位号升序排列所有存活玩家，写入 Redis List，并发布第一个 SPEECH_TURN_EVENT。"""

    async def get_current_speaker(self) -> Optional[str]:
        """获取当前应该发言的玩家 ID（队列头部，但不移除）。"""

    async def submit_speech(self, action: AgentAction, roles: Dict[str, BaseRole], current_phase: GamePhase) -> bool:
        """校验发言顺序，发布 SPEECH_EVENT，弹出队列头部，并发布下一个 SPEECH_TURN_EVENT。"""

    async def skip_current_speaker(self, current_round: int, phase: GamePhase) -> Optional[str]:
        """跳过当前发言玩家（强制弹出队列头部），用于超时场景。"""

    async def is_queue_empty(self) -> bool:
        """检查发言队列是否为空（所有玩家已发言完毕）。"""
```

### 2.2 事件系统扩展

为了支持顺序唤醒，我们在 `EventType` 中新增了 `SPEECH_TURN_EVENT`。

- **`SPEECH_TURN_EVENT`**: 轮到某位玩家发言时发布。
  - **Payload**: `{"player_id": "...", "phase": "...", "round": 1}`
  - **作用**: 任务分发器监听到此事件后，仅为 `player_id` 对应的玩家派发决策任务。

## 3. 调度逻辑与流程流转

### 3.1 GameEngine 集成

在 `GameEngine` 中，发言阶段的流转逻辑如下：

1. **进入发言阶段**:
   在 `advance_phase` 中，如果 `next_phase` 属于 `SPEECH_PHASES`，则调用 `speech_manager.init_queue()`。
2. **动作提交**:
   在 `submit_action` 中，如果当前是发言阶段，将 `SPEAK` 动作路由给 `SpeechManager` 处理。
3. **提前结束检测**:
   在 `_check_early_termination` 中，如果当前是发言阶段，调用 `speech_manager.is_queue_empty()`。如果队列为空，则取消定时器并立即推进到下一阶段。

### 3.2 任务分发器 (Dispatcher) 集成

在 `ai_werewolf_core/tasks/dispatch.py` 中：

1. **取消并发分发**:
   在 `on_phase_transition` 中，当进入 `DAY_DISCUSSION` 等发言阶段时，不再将所有存活玩家加入 `dispatch_targets` 进行并发分发。
2. **新增顺序分发**:
   新增 `on_speech_turn` 监听器，专门监听 `SPEECH_TURN_EVENT`。当事件触发时，仅为 `payload["player_id"]` 对应的玩家派发 `run_agent_decision` 任务。

## 4. 完整发言流程图

```text
[GameEngine] advance_phase() 推进到 DAY_DISCUSSION
       |
       v
[SpeechManager] init_queue()
       |-- 1. 过滤存活玩家并按座位号升序排序
       |-- 2. 写入 Redis List (werewolf:speech_queue:{game_id})
       |-- 3. 发布 SPEECH_TURN_EVENT(player_1)
       |
       v
[Dispatcher] on_speech_turn() 监听到事件
       |-- 派发 player_1 的 Celery 决策任务
       |
       v
[Agent] player_1 思考并提交 SPEAK 动作
       |
       v
[API/Engine] submit_action_internal() -> SpeechManager.submit_speech()
       |-- 1. 校验 player_1 是否在队列头部
       |-- 2. 发布 SPEECH_EVENT (公开广播发言内容)
       |-- 3. Redis LPOP 弹出 player_1
       |-- 4. 获取下一个玩家 (player_2)
       |-- 5. 发布 SPEECH_TURN_EVENT(player_2)
       |
       v
[Dispatcher] on_speech_turn() 监听到事件
       |-- 派发 player_2 的 Celery 决策任务
       |
      ... (循环直至队列为空)
       |
       v
[Agent] 最后一个玩家提交发言
       |
       v
[SpeechManager] submit_speech() 弹出最后一个玩家，队列为空
       |
       v
[GameEngine] _check_early_termination()
       |-- 检测到 is_queue_empty() == True
       |-- 取消当前阶段的超时定时器
       |-- 调用 advance_phase() 推进到 DAY_VOTE
```

## 5. 异常处理与容错

1. **玩家死亡/断线**: 如果轮到的玩家已死亡（例如在发言阶段被毒死，虽然标准规则中少见），`submit_speech` 会在校验时抛出异常。引擎可以通过 `skip_current_speaker` 强制跳过。
2. **超时未发言**: 如果玩家在规定时间内未提交发言，Celery 定时任务触发 `advance_phase_task` 时，引擎可以检测到队列未空，此时可以强制清空队列并推进阶段，或者调用 `skip_current_speaker` 跳过当前玩家。
3. **并发提交**: 如果非当前轮次的玩家恶意提交发言，`SpeechManager` 会通过 `get_current_speaker` 校验并拒绝该动作，保证顺序不被破坏。
