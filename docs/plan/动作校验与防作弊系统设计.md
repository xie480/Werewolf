# 动作校验与防作弊系统设计（修订版）

## 修订说明

**问题**：原设计将"存活校验"作为集中式通用规则——要求行动者必须在 Redis BitMap 中标记为存活。但猎人必须在死亡状态下才能使用技能（HUNTER_SHOOT），一刀切的"必须存活"规则会阻断猎人的唯一合法动作。

**根因**：生存状态要求不是通用的，而是**角色相关的（role-dependent）**。不同角色对同一个动作的生存状态要求可能完全相反。

**解决方案**：采用**角色声明 + 校验器执行**模式。每个角色声明其对每种动作类型的生存状态要求（必须存活 / 必须死亡 / 无要求），集中式 ActionValidator 根据角色声明来执行具体的 BitMap 校验逻辑。

---

## 1. 架构定位与设计目标

在狼人杀 AI 博弈平台中，**Agent 由大语言模型驱动，天然具有不可预测性**。LLM 可能在黑夜试图发言、已死亡的玩家可能试图投票、狼人可能试图在同一夜连续刀多人——这些行为一旦穿透到 Engine 层，会导致游戏状态混乱甚至崩溃。

为了解决这一核心矛盾，本模块承担**游戏入口守卫**的角色，在执行任何 Agent 动作之前构建一道独立于业务逻辑的纯规则防火墙：

- **防作弊 (Anti-Cheat)**：实时检测非法行为（超频提交、跨阶段越权、幽灵玩家行动），拒绝写入并记录异常。
- **动作标准化 (Action Normalization)**：将 LLM 的模糊输出（如 "我投票给3号"）转换为结构化的 `AgentAction`，解析失败时返回明确的拒绝原因供 LLM 自我纠正。
- **速率限制 (Rate Limiting)**：防止 Agent 在短时间内洪水式提交动作（如大量无效 JSON / 拒绝服务式请求），保护 Game Engine 和 Redis 的稳定性。
- **审计日志 (Audit Trail)**：所有被拒绝的动作完整记录（含原始输入、拒绝原因、时间戳），用于评测 Agent 行为质量和调试提示词。

**关键设计原则**：本模块不做游戏规则判定（那是 Role System 和 Manager 的职责），只做**合法性边界检查**。两者互补：
- 防作弊系统回答："这个动作是否可能合法？"（结构完整性、阶段匹配、频率检查、角色声明的生存状态约束）
- Role/Manager 回答："这个动作在此阶段是否确实合法？"（角色权限、目标逻辑、技能次数）

**生存状态校验的特殊处理**：生存状态约束（"必须存活"或"必须死亡"）不是通用规则，而是由具体角色声明。集中校验器仅负责根据角色声明去执行 Redis BitMap 检查——规则由角色定义，校验由防火墙执行。详见 [2.5 节](#25-生存状态声明-get_survival_requirement)。

**文件路径**: `ai_werewolf_core/core/action/`

## 2. 核心组件设计

### 2.1 生存状态需求枚举 (SurvivalRequirement)

**文件路径**: `ai_werewolf_core/schemas/enums.py`（新增枚举）

**Why**：集中校验器需要知道"当前角色对此动作要求存活还是死亡"，但不能在 Validator 中硬编码角色逻辑。通过枚举让每个角色声明自己的需求，Validator 只负责查 BitMap 并比对。

```python
class SurvivalRequirement(str, Enum):
    """角色对动作的生存状态要求。

    用于 ActionValidator 的生存状态校验环节：
    - MUST_BE_ALIVE → Validator 拒绝已死亡的玩家
    - MUST_BE_DEAD  → Validator 拒绝仍存活的玩家
    - ANY           → 不校验生存状态（如 PASS 动作）
    """
    MUST_BE_ALIVE = "MUST_BE_ALIVE"
    MUST_BE_DEAD = "MUST_BE_DEAD"
    ANY = "ANY"
```

| 值 | 含义 | 典型场景 |
|---|---|---|
| `MUST_BE_ALIVE` | 行动者必须在 BitMap 中标记为存活 | SPEAK、VOTE、WOLF_KILL、SEER_CHECK 等 |
| `MUST_BE_DEAD` | 行动者必须在 BitMap 中标记为死亡 | HUNTER_SHOOT（猎人死亡开枪） |
| `ANY` | 不校验生存状态 | PASS（Agent 主动放弃行动，无论死活都允许声明） |

### 2.2 角色基类新增生存状态声明

**文件路径**: `ai_werewolf_core/core/engine/roles/base.py`（新增方法）

在 `BaseRole` 中新增 `get_survival_requirement()` 方法，提供默认实现（大部分动作要求存活），子类可覆盖。

```python
def get_survival_requirement(
    self, action_type: ActionType
) -> SurvivalRequirement:
    """返回此角色对指定动作类型的生存状态要求。

    集中式 ActionValidator 在生存状态校验环节调用此方法，
    获取角色声明的约束，然后去 Redis BitMap 验证实际状态。

    默认行为：PASS 动作不校验生存状态，其余动作要求存活。
    子类覆盖此方法以声明不同的约束（如猎人 HUNTER_SHOOT 要求死亡）。

    Args:
        action_type: 待校验的动作类型。

    Returns:
        SurvivalRequirement 枚举值。
    """
    if action_type == ActionType.PASS:
        return SurvivalRequirement.ANY
    return SurvivalRequirement.MUST_BE_ALIVE
```

**HunterRole 覆盖**：

```python
def get_survival_requirement(
    self, action_type: ActionType
) -> SurvivalRequirement:
    if action_type == ActionType.HUNTER_SHOOT:
        return SurvivalRequirement.MUST_BE_DEAD
    return super().get_survival_requirement(action_type)
```

**Why (此设计优于替代方案)**：

| 方案 | 描述 | 问题 |
|------|------|------|
| ~~A: Validator 硬编码全部规则~~ | 在 Validator 中维护 `Role → ActionType → 存活要求` 映射表 | Validator 耦合角色知识，新增角色需改 Validator |
| ~~B: 删除集中式存活校验~~ | Validator 不查存活，全部交给 Manager 的 `validate_action()` | 失去防火墙意义——死亡玩家的无效请求仍会穿透到 Manager，浪费资源 |
| **C: 角色声明 + Validator 执行（选用）** | 角色提供 `get_survival_requirement()`，Validator 据此执行 BitMap 校验 | ✅ 规则由角色拥有，校验由防火墙执行，职责分离清晰 |

方案 C 遵循**开闭原则**：新增角色时只需覆盖 `get_survival_requirement()`，Validator 无需修改。这符合设计文档原有的设计原则——"Validator 不做游戏规则判定，只做合法性边界检查"——生存状态约束的**规则定义**属于角色，**规则执行**属于防火墙。

### 2.3 动作校验器 (ActionValidator)

**文件路径**: `ai_werewolf_core/core/action/validator.py`

**核心职责**（修订后）：
- **结构化校验**：通过 Pydantic 模型校验动作的字段完整性（`actor_id`、`action_type`、`target_id` 等）。
- **阶段校验**：动作声明的 `phase` 必须与 PhaseStateMachine 的当前阶段一致。
- **生存状态校验**：从角色对象获取 `SurvivalRequirement`，据此到 Redis BitMap 验证实际存活/死亡状态。
- **冷却校验**：同一玩家在同一阶段不可重复提交同类型动作（防止 LLM loop 重复调用工具）。

**Why (独立校验层而非内嵌到 Manager)**:
目前 `ActionResolver.submit_action()`、`VoteManager.submit_vote()` 和 `SpecialActionResolver.handle_action()` 各自在方法开头做校验，存在明显问题：
1. **校验逻辑重复**：三个 Manager 都各自实现了存活检查和阶段检查，代码分散且有细微差异。
2. **新增作弊规则需要改三个地方**：如果在未来想增加"同一阶段最多重试 3 次的限制"，三个 Manager 都要改。
3. **无法统一审计**：被拒绝的动作分散在三个 Manager 的日志中，无法做全局分析。

独立校验层将所有边界检查集中在 `action/` 目录下，Manager 只需关心"校验通过后做什么"。

**核心接口设计**:
```python
class ActionValidator:
    """动作校验器 —— 纯规则防火墙，不依赖 LLM 判定，不做游戏规则判定。"""

    def __init__(self, game_id: str):
        self.game_id = game_id
        self._recent_actions: dict[str, float] = {}  # 冷却追踪
        self._rejected_log: list[RejectedAction] = []  # 拒绝审计日志

    async def validate(
        self,
        action: AgentAction,
        role: BaseRole,                  # 新增：角色对象，用于获取生存状态声明
        current_phase: GamePhase,
    ) -> ValidationResult:
        """
        执行完整校验链：结构化 → 阶段 → 生存状态 → 冷却。
        全部通过返回 ValidationResult.passed()，任一步失败返回 ValidationResult.rejected(reason)。

        生存状态校验环节：
        1. 调用 role.get_survival_requirement(action.action_type) 获取声明
        2. 若声明为 ANY，跳过
        3. 若声明为 MUST_BE_ALIVE，查 Redis BitMap 确认存活
        4. 若声明为 MUST_BE_DEAD，查 Redis BitMap 确认死亡
        """
```

**校验链执行顺序**:
1. **结构化校验** → 最快失败（字段缺失/类型错误），直接在入口拒绝，无需访问 Redis。
2. **阶段校验** → 对比动作声明阶段与当前阶段。
3. **生存状态校验** → 查询角色声明 → 按声明去 Redis BitMap 验证实际状态，O(1) 操作。
4. **冷却校验** → 检查内存中的 `_recent_actions` 字典（同一玩家/阶段/动作类型组合的时间戳）。

**冷却策略**:
- 使用内存字典 `actor_id:phase:action_type → timestamp` 追踪。
- 默认冷却窗口：同一玩家同一阶段同一动作类型 2 秒内不可重复提交。
- **Why**: 防止 LLM 在 `while` 循环中重复调用提交工具（如 LangGraph 的 tool calling 循环），打爆 Redis 和 EventBus。

### 2.4 防作弊检测器 (AntiCheatDetector)

**文件路径**: `ai_werewolf_core/core/action/anti_cheat.py`

**核心职责**:
- **超时检测**：如果某个阶段超过配置的最大时长（如 NIGHT_WOLF_ACT 默认 60 秒），标记所有未行动的强制 PASS。
- **幽灵玩家检测**：检测不存在于当前对局的 `actor_id`（可能是 LLM 幻觉或伪造请求）。
- **跨阵营越权**：狼人不可在狼人频道外提交 WOLF_KILL（FACTION 级别权限的基础校验）。
- **重放攻击防护**：检测重复提交完全相同的动作（相同 actor_id + action_type + target_id + 时间窗口内）。

**Why (独立于 Validator 的原因)**:
校验器 (Validator) 回答"这个动作是否合法"，防作弊器 (Detector) 回答"这个 Agent 是否诚信"。
两者的关注点和响应方式不同：
- 校验失败 → 返回拒绝原因，Agent 可以纠正后重试（正常的交互流程）。
- 作弊检测触发 → 记录 WARNING 级别事件，可能降级该 Agent 的行为（如强制 PASS、限制后续提交频率）。

**核心接口设计**:
```python
class AntiCheatDetector:
    """防作弊检测器 —— 检测异常行为模式并触发降级措施。"""

    def __init__(self, game_id: str, event_bus: EventBus):
        self.game_id = game_id
        self.event_bus = event_bus
        self._action_counts: dict[str, int] = {}        # action_count[actor_id] = 累计提交次数
        self._duplicate_hashes: set[str] = set()         # 重放检测哈希集合
        self._violations: list[ViolationRecord] = []     # 违规记录

    async def inspect(
        self,
        action: AgentAction,
        roles: dict[str, BaseRole],
    ) -> InspectionResult:
        """执行防作弊检测：幽灵玩家 → 越权 → 重放 → 超频。"""

    def is_rate_limited(self, actor_id: str) -> bool:
        """检查指定玩家是否已被限速（超频阈值触发后）。"""

    def apply_penalty(self, actor_id: str, penalty: PenaltyType) -> None:
        """对违规玩家施加惩罚：强制 PASS / 限速 / 标记为可疑。"""
```

**检测规则与惩罚**:

| 规则 | 检测条件 | 惩罚 |
|------|----------|------|
| 幽灵玩家 | `actor_id` 不在 `roles` 中 | 拒绝动作，发布 SECURITY_ALERT 事件 |
| 跨阵营越权 | 狼人在非夜间阶段提交 WOLF_KILL | 拒绝动作，记录违规 |
| 重放攻击 | 相同动作哈希在 30 秒内重复 | 拒绝重复，警告日志 |
| 超频提交 | 同一玩家单阶段提交超过 10 次 | 限速 5 秒，后续动作强制延迟处理 |

**违规记录持久化**: 所有 `ViolationRecord` 在阶段结算或对局结束时通过 EventBus 发布为 `SECURITY_ALERT` 事件，进入 EventRecord 冷数据表，供复盘分析。

### 2.5 审计日志 (AuditLogger)

**文件路径**: `ai_werewolf_core/core/action/audit.py`

**核心职责**:
- 记录**所有**被拒绝的动作（无论原因），包含原始输入、拒绝原因、时间戳。
- 提供按 `actor_id` 聚合的拒绝统计，用于衡量 Agent 行为质量（评测系统输入之一）。
- 在 DEBUG 模式下输出完整的动作提交链，便于追踪 LLM 的决策轨迹。

**Why (独立于 structlog 的审计日志)**:
structlog 是通用日志框架，其结构化输出面向人类运维。审计日志则需要面向评测系统 (Evaluation System) 和自动告警——需要明确的 schema、可查询的存储、以及可量化的聚合指标。

**核心接口设计**:
```python
@dataclass(frozen=True)
class RejectedAction:
    """被拒绝的动作记录 —— 不可变值对象。"""
    action: AgentAction
    reason: str
    rejector: str          # "ActionValidator" / "AntiCheatDetector" / "RoleSystem"
    timestamp: datetime
    actor_role: Optional[Role] = None

class AuditLogger:
    """审计日志记录器 —— 记录所有拒绝事件，供评测和排障。"""

    def __init__(self, game_id: str):
        self.game_id = game_id
        self.rejected: list[RejectedAction] = []
        self.accepted_count: dict[str, int] = {}  # actor_id → 通过次数

    def log_rejection(self, rejected: RejectedAction) -> None:
        """记录一次拒绝事件。"""

    def log_acceptance(self, actor_id: str) -> None:
        """记录一次通过事件。"""

    def get_rejection_stats(self) -> dict[str, int]:
        """按 actor_id 聚合的拒绝次数统计，用于评测 Agent 质量。"""

    def snapshot(self) -> AuditSnapshot:
        """生成当前审计快照，可用于发布到 EventBus 或存入复盘数据。"""
```

### 2.6 动作门控 (ActionGate)

**文件路径**: `ai_werewolf_core/core/action/gate.py`

**核心职责**（修订后）:
- 作为 `core/action/` 模块的统一入口，编排 Validator → Detector → Audit 的完整管道。
- **Validator 阶段需要传入对应的 role 对象**以支持角色声明的生存状态校验。
- 对 Game Engine 暴露最简洁的接口：`await action_gate.admit(action, role)` → 通过/拒绝。

**Why (Facade 模式)**:
GameEngine 不需要知道内部有多少个校验组件，只需要一个 `Admit/Reject` 的二值结果。Gate 封装了所有校验和审计逻辑，单点调用即可。

**核心接口设计**:
```python
class ActionGate:
    """动作门控 —— 动作校验与防作弊系统的统一入口 (Facade)。

    编排完整的校验管道：
    1. ActionValidator.validate()     —— 基本合法性校验（含角色声明的生存状态校验）
    2. AntiCheatDetector.inspect()    —— 防作弊检测
    3. AuditLogger.log_*()            —— 审计记录
    返回最终的二值结果：Admit 或 Reject（含原因）。
    """

    def __init__(self, game_id: str, event_bus: EventBus):
        self.game_id = game_id
        self.validator = ActionValidator(game_id)
        self.anti_cheat = AntiCheatDetector(game_id, event_bus)
        self.audit = AuditLogger(game_id)

    async def admit(
        self,
        action: AgentAction,
        role: BaseRole,                     # 单个角色对象（替代原来的 roles dict）
        roles: dict[str, BaseRole],         # 所有角色映射（供 Detector 使用）
        current_phase: GamePhase,
    ) -> AdmitResult:
        """
        动作准入判断 —— 通过全部校验链方可放行。

        校验链（短路求值，任一步失败即停止）:
        1. ActionValidator.validate(action, role, current_phase)
           → 失败则返回 Rejected
           （Validator 内部根据 role.get_survival_requirement() 做生存状态校验）
        2. AntiCheatDetector.inspect(action, roles)
           → 失败则记录违规并返回 Rejected
        3. AuditLogger.log_acceptance(actor_id)
           → 通过

        返回 AdmitResult(admitted=True/False, reason=...)
        """

    def get_rejection_stats(self) -> dict[str, int]:
        """按 actor_id 聚合的拒绝统计。"""
```

## 3. Game Engine 集成方式

在 `GameEngine` 的 `submit_action` 方法中，`ActionGate` 处于入口第一层，传入具体角色对象以支持生存状态校验：

```python
async def submit_action(self, action: AgentAction) -> SubmitResult:
    # 获取行动者的角色对象
    role = self.roles.get(action.actor_id)
    if role is None:
        return SubmitResult.rejected("未知玩家")

    # 1. 动作门控（纯规则防火墙，不做游戏逻辑判定）
    admit_result = await self.action_gate.admit(
        action, role, self.roles, await self.state_machine.get_current_phase()
    )
    if not admit_result.admitted:
        return SubmitResult.rejected(admit_result.reason)

    # 2. 路由到具体 Manager（游戏规则校验 + 业务逻辑）
    phase = await self.state_machine.get_current_phase()
    if phase in night_phases:
        self.resolver.submit_action(action, self.roles, phase)
    elif phase in vote_phases:
        await self.vote_manager.submit_vote(action, self.roles, phase)
    elif phase == GamePhase.HUNTER_SHOOT:
        result = await self.special_action_resolver.handle_action(action, self.roles, phase)
    # ...
```

**为什么 Gate 在 Manager 之前**：
- Gate 是纯规则防火墙，可以做乐观拒绝（如"阶段不匹配"、"生存状态不满足角色声明"），节省 Manager 的校验开销。
- Gate 的拒绝可能是可恢复的（Agent 纠正参数后重试），Manager 的拒绝通常是不可恢复的（角色无此权限）。
- Gate 统计所有拒绝事件，Manager 只需关心业务逻辑，审计职责完全外移。

**生存状态校验的职责边界**:
- **Gate 层**：根据 `role.get_survival_requirement(action_type)` 的声明，去 Redis BitMap 验证实际状态是否匹配。如果猎人声明 HUNTER_SHOOT 需要死亡状态，但 Redis BitMap 显示猎人仍存活 → Gate 拒绝。
- **Manager 层**：`role.validate_action(phase, action_type)` 仍然执行，做更深层的角色专属校验（如女巫解药只能用一次、预言家每夜只能验一人等）。这是游戏规则判定，不属于防火墙。

## 4. 角色生存状态需求矩阵

| 角色 | 动作类型 | SurvivalRequirement | 原因 |
|---|---|---|---|
| 村民 | SPEAK | MUST_BE_ALIVE | 死亡村民不能发言 |
| 村民 | VOTE | MUST_BE_ALIVE | 死亡村民不能投票 |
| 村民 | PASS | ANY | 无论死活都可以声明放弃行动 |
| 狼人 | WOLF_KILL | MUST_BE_ALIVE | 存活狼人才能刀人 |
| 预言家 | SEER_CHECK | MUST_BE_ALIVE | 存活预言家才能验人 |
| 女巫 | WITCH_SAVE | MUST_BE_ALIVE | 存活女巫才能用解药 |
| 女巫 | WITCH_POISON | MUST_BE_ALIVE | 存活女巫才能用毒药 |
| **猎人** | **HUNTER_SHOOT** | **MUST_BE_DEAD** | **猎人必须死亡才能开枪** |

## 7. 验证方法

1. **单元测试**：`test_validator.py` 中必须包含以下关键场景：
   - 存活村民提交 SPEAK → 通过
   - 死亡村民提交 SPEAK → 拒绝（reason 包含 "角色要求存活但玩家已死亡"）
   - **死亡猎人提交 HUNTER_SHOOT → 通过**（关键回归用例）
   - **存活猎人提交 HUNTER_SHOOT → 拒绝**（reason 包含 "角色要求死亡但玩家仍存活"）
   - 任意角色提交 PASS → 通过（不校验生存状态）
2. **集成测试**：在完整对局中模拟猎人被投票放逐 → HUNTER_SHOOT 阶段 → 猎人提交开枪动作 → 验证 Gate 放行且 Manager 正常结算。
3. **回归测试**：确保现有狼人/女巫/预言家/村民的所有动作校验行为不变。

## 8. 遵循的架构规范

1. **规则硬编码**: 所有校验规则在 Python 代码中硬编码，不依赖 LLM 判定。
2. **无魔法字符串**: 阶段、动作类型、违规类型、**生存状态需求**等全部使用 `schemas/enums.py` 中的枚举。
3. **强类型声明**: 接口参数和返回值严格使用 Type Hinting 和 `@dataclass` 模型。
4. **异步优先**: 涉及 Redis 操作（生存状态校验）和事件发布的校验步使用 `async/await`。
5. **结构化日志**: 拒绝/违规使用 `structlog` 记录，注入 `game_id`、`actor_id`、`reason`、`survival_requirement` 等上下文。
6. **最小变更原则**: 从现有 Manager 中**移除**边界检查逻辑（而非复制），避免两套校验并存。
7. **开闭原则**: 新增角色时只需覆盖 `get_survival_requirement()`，Validator 和 Gate 无需修改。
8. **注释规范**: 核心方法包含 `**Why**` 注释，边界条件和异常处理原因明确说明。
