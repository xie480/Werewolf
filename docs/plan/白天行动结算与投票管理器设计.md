# 白天行动结算与投票管理器设计

## 1. 架构定位与设计目标

在狼人杀对局中，白天的行动与夜晚有着本质的区别：
- **夜晚行动**（狼刀、女巫用药、预言家验人）具有严格的因果链，需要**串行提交、延迟统一结算**。
- **白天行动**（放逐投票、猎人开枪、白痴翻牌等）通常是**即时结算**或**条件触发**的，不需要复杂的草稿暂存机制。

为了遵循单一职责原则 (SRP) 并保持 `GameEngine` 主流程的清晰，我们将白天的核心逻辑拆分为两个独立的管理器：
1. **投票管理器 (`VoteManager`)**：专职处理群体性的投票行为（放逐投票、PK 投票、警长竞选投票）。
2. **特殊行动结算器 (`SpecialActionResolver`)**：专职处理白天特殊角色的即时技能插队（猎人开枪、白痴翻牌等）。

## 2. 核心组件设计

### 2.1 投票管理器 (VoteManager)

**文件路径**: `ai_werewolf_core/core/engine/vote_manager.py`

**核心职责**:
- **选票收集与校验**：记录投票意图，校验投票人是否存活、是否在合法的投票阶段、目标是否合法（如 PK 阶段只能投 PK 名单上的人）。
- **计票与平票处理**：统计最高票。如果出现平票，返回平票名单供引擎触发 PK 流程。
- **即时结算**：如果产生唯一最高票，直接调用目标的 `die()` 方法，并通过 EventBus 发布 `PLAYER_DEATH` 事件。

**Redis Hash 无状态存储设计**:

VoteManager 不保存任何投票状态在实例变量中，所有选票数据存储于 Redis Hash：
- **Key**: `werewolf:vote:{game_id}:{round}`
- **Field**: `voter_id`（投票人）
- **Value**: `target_id`（被投人，空字符串表示弃权）
- **TTL**: 24 小时（86400 秒）

**并发安全性**: Redis HSET 天然支持多 Worker 并发写入同一 Hash 的不同 Field，写入同一 Field 时最后写入的值生效（遵循"最终一次投票为准"策略）。

**即时结算 vs 延迟结算**: 白天投票的结果立即可知且不可逆转，不存在像夜晚那样"狼刀 → 女巫救 → 女巫毒"的因果链反转，因此采用即时结算模式。

**核心接口设计**:
```python
class VoteManager:
    def __init__(self, game_id: str, event_bus: EventBus):
        self.game_id = game_id
        self.event_bus = event_bus
        self._redis = None            # Redis 客户端（懒初始化）
        self.pk_candidates = None     # PK 候选人名单（控制面参数）
        self._current_round = 0       # 当前投票轮次
        self._player_status = PlayerStatusManager()  # 存活状态同步

    def begin_vote(self, round_num: int, pk_candidates=None) -> None:
        """开启新一轮投票，设置轮次和 PK 候选人。"""

    async def submit_vote(self, action, roles, current_phase) -> bool:
        """校验投票合法性后，通过 HSET 原子写入 Redis Hash。"""

    async def resolve_vote(self, roles, current_round=0) -> VoteResolveResult:
        """HGETALL 拉取全量选票，计票统计，唯一最高票执行即时死亡结算。"""
```

**死亡结算流程**:
1. 调用 `target_role.die()` 标记角色内存状态
2. 调用 `PlayerStatusManager.mark_dead()` 同步更新 Redis BitMap (`werewolf:alive:{game_id}`)
3. 通过 EventBus 发布 `PLAYER_DEATH` 事件（含死亡原因、角色类型、阵营等）

**Redis 操作重试策略**: 关键写操作（HSET, HGETALL）支持最多 3 次指数退避重试（0.1s, 0.2s, 0.3s），确保瞬时网络抖动不丢失选票。Redis 不可用时拒绝投票操作，不允许降级。

### 2.2 特殊行动结算器 (SpecialActionResolver)

**文件路径**: `ai_werewolf_core/core/engine/special_action_resolver.py`

**核心职责**:
- **条件校验**：判断当前是否处于允许特殊行动的阶段（如 `HUNTER_SHOOT`），以及角色是否满足发动条件（如猎人已死亡、未被毒杀、每局仅可开枪一次）。
- **即时结算**：校验通过后立刻执行状态变更（如目标死亡），不等待阶段结束。
- **事件发布**：技能执行后通过 EventBus 发布死亡事件，并同步更新 Redis BitMap 存活状态。

**当前支持的特殊行动**:
- `HUNTER_SHOOT`: 猎人死亡时开枪带走一名存活玩家。触发条件：猎人已死、非毒杀、本局尚未开过枪。
- `PASS`（猎人放弃开枪）: 猎人在 `HUNTER_SHOOT` 阶段可选择放弃。

**核心接口设计**:
```python
class SpecialActionResolver:
    def __init__(self, game_id: str, event_bus: EventBus):
        self.game_id = game_id
        self.event_bus = event_bus
        self.hunter_has_shot = False   # 猎人开枪次数限制
        self._player_status = PlayerStatusManager()

    async def handle_action(self, action, roles, current_phase) -> SpecialActionResult:
        """路由分发到对应处理器（目前仅支持猎人开枪/PASS）。"""
```

**返回结果**: `SpecialActionResult` 包含 `was_handled`、`deaths_caused`、`next_phase_hint`，供引擎决定后续阶段流转。

## 3. GameEngine 调度逻辑

在 `GameEngine` 中，根据当前阶段将动作路由到不同的管理器：

```python
async def submit_action(self, action: AgentAction):
    phase = await self.state_machine.get_current_phase()

    # 1. 夜晚行动 -> 交给 ActionResolver (延迟结算)
    if phase in night_phases:
        self.resolver.submit_action(action, self.roles, phase)

    # 2. 投票行动 -> 交给 VoteManager (收集选票 → Redis Hash)
    elif phase in (GamePhase.DAY_VOTE, GamePhase.DAY_PK_VOTE):
        await self.vote_manager.submit_vote(action, self.roles, phase)

    # 3. 特殊行动 -> 交给 SpecialActionResolver (即时结算)
    elif phase == GamePhase.HUNTER_SHOOT:
        result = await self.special_action_resolver.handle_action(action, self.roles, phase)
        # 引擎根据 result.next_phase_hint 推进状态机

    # 4. 其他常规行动 (如发言)
    elif phase in (GamePhase.DAY_DISCUSSION, GamePhase.LAST_WORDS):
        self.handle_speak(action)
```

## 4. 遵循的架构规范 (agent.md)

1. **规则硬编码**: 投票平票逻辑、猎人开枪条件等全部在 Python 代码中硬编码，不依赖 LLM 判定。
2. **无魔法字符串**: 所有阶段 (`GamePhase`)、动作类型 (`ActionType`) 均使用 `schemas/enums.py` 中的枚举。
3. **强类型声明**: 接口参数和返回值严格使用 Type Hinting 和 Pydantic/DataClass 模型（如 `VoteResolveResult`、`SpecialActionResult`）。
4. **异步优先**: 结算过程中涉及事件发布 (`event_bus.publish`) 和 Redis 操作，全部使用 `async/await`。
5. **结构化日志**: 使用 `structlog` 记录投票明细、平票情况、特殊行动触发等关键节点，并注入 `game_id` 等上下文。
6. **注释规范**: 核心方法必须包含 `**Why**` 注释，解释设计决策。
7. **Redis 无状态**: VoteManager 和 SpecialActionResolver 不保存游戏状态于实例变量。选票数据存 Redis Hash，存活状态通过 PlayerStatusManager 同步 Redis BitMap，确保多 Worker 一致性。
