# 角色与能力系统设计 (Role & Ability System)

## 1. 模块职责概述
本模块负责定义狼人杀游戏中所有角色的基础属性、阵营归属以及在不同游戏阶段的行动权限。它是 Game Engine 进行动作校验（Action Validation）的基础，确保"死人不能说话"、"平民不能在夜晚杀人"等基本规则得到严格执行。

## 2. 核心架构设计
采用面向对象的设计，定义一个基础的 `BaseRole` 抽象类，所有具体角色（如 `WerewolfRole`, `SeerRole`, `WitchRole`, `VillagerRole`, `HunterRole`）继承自该基类。
- **属性定义**：角色名称、所属阵营（狼人/好人）、座位号（`seat_number`）、是否拥有夜间行动能力。
- **权限校验**：提供 `validate_action(phase, action_type)` 统一入口，先尝试通用动作校验（`can_perform_common_action`），再尝试角色专属校验（`can_act`）。两次校验均失败时拒绝动作。
- **状态管理**：管理角色在当前对局中的状态（存活/死亡、技能是否已使用，如女巫的解药/毒药）。

### 2.1 存活状态的双轨管理

**进程内存 (`is_alive`)**：`BaseRole` 实例变量保持 `is_alive: bool` 作为当前进程内的快速读取路径。所有 `can_act` 和 `can_perform_common_action` 校验均直接读取此属性，避免每次校验都查 Redis。

**Redis BitMap (`PlayerStatusManager`)**：通过 `werewolf:alive:{game_id}` BitMap 实现跨进程存活状态共享。
- 角色死亡时调用链路：`role.die()` → 标记 `is_alive=False` → 调用方（VoteManager / ActionResolver / SpecialActionResolver）调用 `PlayerStatusManager.mark_dead()` 同步更新 Redis BitMap → 异步更新 DB `PlayerRecord`（最终一致性）。
- 角色复活时调用链路（女巫解药）：`role.revive()` → 标记 `is_alive=True` → 调用方调用 `PlayerStatusManager.mark_alive()` 同步更新 Redis BitMap → 异步更新 DB。

**一致性保证**：同一进程内始终以角色实例的 `is_alive` 为准（内存优先）；跨进程 Worker 通过 PlayerStatusManager 读取 Redis BitMap 获取最新存活状态；DB 采用最终一致性异步更新，失败不阻塞游戏流程。

## 3. 接口与方法定义

### 3.1 阵营与角色枚举（已实现在 `schemas/enums.py`）
- `Faction`: `VILLAGER`（好人阵营）、`WEREWOLF`（狼人阵营）
- `Role`: `VILLAGER`, `WEREWOLF`, `SEER`, `WITCH`, `HUNTER`

### 3.2 BaseRole 核心接口

```python
class BaseRole(ABC):
    """角色基类 —— 所有具体角色的抽象父类。"""
    
    role_type: Role      # 子类必须覆盖
    faction: Faction      # 子类必须覆盖
    
    def __init__(self, player_id: str):
        self.player_id = player_id
        self.is_alive: bool = True
    
    # ── 通用动作校验 ──
    def can_perform_common_action(self, phase: GamePhase, action_type: ActionType) -> bool:
        """
        校验通用动作（发言、投票、空过）。
        规则：
        - 死亡玩家不能执行任何动作
        - SPEAK 仅在 DAY_DISCUSSION / DAY_PK_DISCUSSION / LAST_WORDS 合法
        - VOTE 仅在 DAY_VOTE / DAY_PK_VOTE 合法
        - PASS 在所有阶段对所有存活角色合法
        """
    
    # ── 抽象接口 ──
    @abstractmethod
    def can_act(self, phase: GamePhase, action_type: ActionType) -> bool:
        """校验角色专属动作 —— 子类必须实现。"""
    
    # ── 统一校验入口 ──
    def validate_action(self, phase: GamePhase, action_type: ActionType) -> bool:
        """先通用校验 → 再角色专属校验 → 两者均失败时拒绝。"""
    
    # ── 生命周期 ──
    def die(self) -> None:
        """标记角色死亡（设置 is_alive=False）。"""
    
    def revive(self) -> None:
        """复活角色（设置 is_alive=True，女巫解药专用）。"""
```

### 3.3 各角色专属动作窗口

| 角色 | 合法阶段 | 合法动作 | 额外条件 |
|---|---|---|---|
| **狼人 (WerewolfRole)** | `NIGHT_WOLF_ACT` | `WOLF_KILL` | 必须存活 |
| **女巫 (WitchRole)** | `NIGHT_WITCH_ACT` | `WITCH_SAVE`, `WITCH_POISON` | 必须存活 + `has_antidote`/`has_poison` 为 True |
| **预言家 (SeerRole)** | `NIGHT_SEER_ACT` | `SEER_CHECK` | 必须存活 |
| **猎人 (HunterRole)** | `HUNTER_SHOOT` | `HUNTER_SHOOT` | **必须死亡** + 未被毒杀（由 ActionResolver 判定）|
| **平民 (VillagerRole)** | — | 无专属动作 | 仅通用动作 |

### 3.4 物品消费接口（女巫专用）

```python
class WitchRole(BaseRole):
    def __init__(self, player_id: str):
        super().__init__(player_id)
        self.has_antidote: bool = True
        self.has_poison: bool = True
    
    def use_antidote(self) -> None:
        """消费解药，若已使用则抛出 ValueError。"""
    
    def use_poison(self) -> None:
        """消费毒药，若已使用则抛出 ValueError。"""
```

## 4. 关键数据流转过程

### 4.1 角色分配
在 `INIT` 阶段，引擎根据房间配置实例化对应的 `Role` 对象，绑定到具体的 `player_id`。同时通过 `PlayerStatusManager.init_players()` 将玩家身份信息写入 Redis Hash (`werewolf:players:{game_id}`) 和 BitMap (`werewolf:alive:{game_id}`)。

### 4.2 动作校验
1. Agent 提交 `AgentAction`（如 `{"action_type": "WOLF_KILL", "actor_id": "p1", "phase": "NIGHT_WOLF_ACT"}`）。
2. 引擎的 `ActionResolver` 查找 `p1` 对应的 `Role` 实例。
3. 调用 `role.validate_action(current_phase, action_type)`。
4. 先执行 `can_perform_common_action`（通用校验：存活 + 阶段 + 动作类型），再执行 `can_act`（角色专属校验）。
5. 如果返回 `False`，引擎拒绝动作并抛出 `ActionValidationError`。

### 4.3 死亡结算（跨模块协作）
1. 结算模块（VoteManager / ActionResolver / SpecialActionResolver）调用 `target_role.die()` 标记内存状态。
2. 结算模块调用 `PlayerStatusManager.mark_dead(game_id, player_id, seat_number)` 同步 Redis BitMap（SETBIT 0）。
3. `PlayerStatusManager` 异步更新 DB `PlayerRecord.is_alive=False`（最终一致性）。
4. 结算模块通过 EventBus 发布 `PLAYER_DEATH` 事件。

### 4.4 状态更新
如果动作合法并最终结算成功（如女巫成功使用解药），引擎调用 `role.use_antidote()` 更新角色内部状态。

## 5. 依赖关系说明
- **Schemas (`ai_werewolf_core/schemas/enums.py`)**：强依赖。角色类型、阵营、动作类型必须使用预定义的枚举。
- **Action Resolver (`ai_werewolf_core/core/engine/resolver.py`)**：被依赖。解析器在处理动作前调用本模块进行权限校验。
- **PlayerStatusManager (`ai_werewolf_core/utils/player_status.py`)**：被依赖。死亡/复活后由调用方通过此模块同步 Redis BitMap 和 DB。
- **EventBus (`ai_werewolf_core/core/event/bus.py`)**：间接依赖。死亡事件由调用方通过 EventBus 发布，不在角色层处理。
