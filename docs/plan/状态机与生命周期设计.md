# 状态机与生命周期设计 (Phase State Machine & Lifecycle)

## 1. 模块职责概述
本模块是 Game Engine 的核心骨架，负责管理整局狼人杀游戏的生命周期（创建、初始化、销毁）以及对局内各个阶段（白天、黑夜、投票等）的严格流转。它必须是**硬编码的绝对裁判**，不允许 LLM 参与状态流转的决策。

## 2. 核心架构设计
模块分为两个主要部分：
- **Lifecycle Manager (生命周期管理器)**：管理全局状态（`INIT -> START -> RUNNING -> SETTLING -> FINISHED -> ABORTED`）。
- **Phase State Machine (阶段状态机)**：管理 `RUNNING` 状态下的具体游戏阶段流转。

设计原则：
- **状态不可逆与严格校验**：状态流转必须遵循预定义的有向图，非法流转请求直接拒绝。
- **事件驱动**：状态变更通过 Event Bus 发布事件（`PHASE_TRANSITION_EVENT`、`SYSTEM_ANNOUNCEMENT`），解耦下游模块（如前端推送、Agent 唤醒）。
- **Redis 无状态架构**：状态不再保存在实例变量中，而是存储于 Redis Hash，确保多 Worker 进程共享同一份权威状态。

## 3. Redis 上下文存储设计

### 3.1 数据模型
- **Key**: `werewolf:game:{game_id}:context` (Hash)
- **Fields**:
  - `phase`: 当前游戏阶段 (`GamePhase` 枚举值，如 `"DAY_VOTE"`；`None` 或 `"None"` 表示尚未开始)
  - `round`: 当前轮次 (整数，默认 0)
  - `status`: 全局生命周期状态 (`GameStatus` 枚举值，如 `"RUNNING"`)
- **TTL**: 对局结束后 1 小时 (3600 秒)。长时间对局可在关键阶段调用 `PlayerStatusManager.extend_ttl()` 刷新。

### 3.2 读写策略
- **PhaseStateMachine** 读取 `phase` + `round`，更新时使用 `HSET` 原子覆盖。
- **LifecycleManager** 读取 `status`，采用 **Write-Through** 模式：先写 Redis，再同步写 DB `GameRecord`。
- 写入顺序: 校验 → Redis HSET → 同步更新 DB → 发布事件

### 3.3 多 Worker 一致性保障
- 所有 Worker 从 Redis Hash 读取当前阶段，任意 Worker 均可执行 `transition_to()`。
- 状态迁移操作本身不是分布式锁保护的——在投票结算等关键节点，由 Game Engine 的编排逻辑串行调用，不需要锁。

## 4. 全局生命周期状态机 (LifecycleManager)

**文件路径**: `ai_werewolf_core/core/engine/lifecycle.py`

**合法状态迁移**:
| 当前状态 | 后继状态 |
|---|---|
| INIT | START |
| START | RUNNING, ABORTED |
| RUNNING | SETTLING, ABORTED |
| SETTLING | FINISHED, ABORTED |
| FINISHED | (无) |
| ABORTED | (无) |

**核心方法**:
- `init_game()`: 初始化对局，写入 DB `GameRecord`，设置 Redis 上下文 (`status=INIT`)。
- `start_game()`: 校验 `INIT → START`，分配身份，初始化 `PhaseStateMachine`，迁移到 `START → RUNNING`。
- `end_game(winner_faction)`: 校验 `RUNNING → SETTLING → FINISHED`，发布结算事件，设置 Redis TTL。
- `abort_game(reason)`: 任意非终态可中止，记录原因，设置 TTL。

**Write-Through 流程** (`_set_status`):
1. 从 Redis 读取当前状态
2. 校验 `VALID_STATUS_TRANSITIONS` 合法性
3. `HSET` 更新 Redis Hash
4. `UPDATE GameRecord SET status=...` 同步写 DB（失败不阻塞，记录 ERROR 日志）
5. 发布 `SYSTEM_ANNOUNCEMENT` 事件

## 5. 阶段状态机 (PhaseStateMachine)

**文件路径**: `ai_werewolf_core/core/engine/state_machine.py`

**完整阶段迁移图**:

```
None → INIT
INIT → NIGHT_START

NIGHT_START → NIGHT_WOLF_ACT → NIGHT_WITCH_ACT → NIGHT_SEER_ACT → NIGHT_RESOLVE
NIGHT_RESOLVE → DAY_START

DAY_START → DAY_DISCUSSION | HUNTER_SHOOT | LAST_WORDS | GAME_OVER
DAY_DISCUSSION → DAY_VOTE
DAY_VOTE → VOTE_RESOLVE | DAY_PK_DISCUSSION

VOTE_RESOLVE → HUNTER_SHOOT | LAST_WORDS | NIGHT_START | GAME_OVER

DAY_PK_DISCUSSION → DAY_PK_VOTE
DAY_PK_VOTE → VOTE_RESOLVE

HUNTER_SHOOT → LAST_WORDS | DAY_DISCUSSION | GAME_OVER
LAST_WORDS → DAY_DISCUSSION | NIGHT_START

GAME_OVER → INIT | None (彻底结束)
```

**轮次递增规则**: 仅当目标阶段为 `NIGHT_START` 时，`round` 自增 1，标记新的"天黑-天亮"循环。

**迁移原子性**: Phase 迁移成功 = Redis HSET 成功 + EventBus publish 成功。若 Redis 写入失败则整个迁移中止并抛出异常。

## 6. 关键数据流转过程
1. **初始化**：API 接收创建房间请求，调用 `LifecycleManager.init_game()`，Redis 写入 `status=INIT, phase=None, round=0`。
2. **开局**：分配身份完毕，调用 `start_game()`，状态变迁 `INIT → START → RUNNING`，`PhaseStateMachine` 迁移到 `INIT → NIGHT_START`（轮次变为 1）。
3. **阶段流转**：
   - 引擎内部逻辑（如夜晚行动全部提交完毕，或倒计时结束）触发 `transition_to(DAY_START)`。
   - 状态机从 Redis 加载当前阶段，校验 `VALID_TRANSITIONS` 合法性。
   - 校验通过后 `HSET` 更新 Redis，记录结构化日志。
   - 发布 `PHASE_TRANSITION_EVENT` 事件（`visibility=PUBLIC`），payload 含 `old_phase`、`new_phase`、`round`。
   - 监听该事件的模块（如 Agent Runtime）被唤醒，开始执行白天逻辑。

## 7. 依赖关系说明
- **Event Bus (`ai_werewolf_core/core/event/bus.py`)**：强依赖。状态机不直接调用 Agent 或前端 API，而是通过发布 `PHASE_TRANSITION_EVENT`、`SYSTEM_ANNOUNCEMENT` 事件驱动外部系统。
- **Redis Client (`ai_werewolf_core/utils/redis_client.py`)**：强依赖。所有状态读写通过 RedisClientManager 共享连接池。Redis 不可用时抛出 `RedisUnavailableException`，状态迁移操作中止。
- **Database (`ai_werewolf_core/db/models.py`)**：Write-Through 模式同步更新 `GameRecord.status`。
- **Logger (`ai_werewolf_core/utils/logger.py`)**：强依赖。每次状态流转记录结构化日志，包含 `game_id`、`old_phase`、`new_phase`、`round`、`old_status`、`new_status`。
