# 事件总线 (Event Bus) 架构设计与实施计划

## 1. 架构概述 (Architecture Overview)

根据 `docs/agent.md` 和 `docs/system/Event System.md` 的规范，AI 狼人杀系统采用 **事件溯源 (Event Sourcing)** 架构。
**Event Bus (事件总线)** 是整个系统的“神经中枢”。所有的状态变化（如阶段切换、玩家死亡、发言、投票）都必须封装为 `Event` 对象，通过 Event Bus 进行分发和记录。

**核心原则：**
*   **不可篡改的事实**：Event 代表已经发生的既定事实（Fact），一旦产生不可修改。
*   **双轨制混合架构 (Hybrid Architecture)**：
    *   **对内 (Push/Pub-Sub)**：针对游戏引擎内部组件（如日志、数据库持久化、状态机、投票管理器），采用内存级别的发布/订阅模式，通过回调函数直接推送，保证极低的延迟和高效的内部状态流转。
    *   **对外 (Pull/MQ Stream)**：针对外部的 AI Agent（玩家），采用基于 Redis Stream 的消息队列模式。AI Agent 不参与内部的推送循环，而是根据自身节奏主动拉取（Pull）事件。这实现了引擎与 AI 的解耦，防止 AI 思考耗时阻塞游戏主流程，并天然支持分布式部署和断线重连。
*   **严格的可见性隔离**：必须根据 `Visibility` (PUBLIC, PRIVATE, FACTION) 严格控制事件的路由，防止信息泄露（防剧透）。
*   **严格的时序保证**：每个事件必须拥有全局单调递增的 `seq_num`，防止并发导致的时序错乱和 AI 幻觉。

## 2. 核心组件设计 (Core Components)

### 2.1 Event 模型 (已在 `schemas/models.py` 中定义)
复用现有的 `Event` Pydantic 模型，确保强类型校验。
包含 `event_id`, `game_id`, `seq_num`, `event_type`, `visibility`, `target_agents`, `timestamp`, `payload`。

### 2.2 EventBus 类 (核心调度器)
位于 `ai_werewolf_core/core/event/bus.py`。
*   **职责**：接收事件、分配 `seq_num`、写入 Redis Stream 热数据缓存、异步持久化到 PostgreSQL、在内存中路由分发给内部订阅者。
*   **核心方法**：
    *   `publish(event: Event)`: 发布事件。执行双轨操作：先写入 Redis Stream，然后通过 `for` 循环分发给内存中的内部订阅者。
    *   `subscribe(event_type: EventType, handler: Callable)`: 内部组件订阅特定类型的事件（如状态机订阅阶段切换）。**注意：AI Agent 绝不能使用此方法订阅。**
    *   `subscribe_all(handler: Callable)`: 内部组件订阅所有事件（如全局 Logger 或 DB 持久化器）。
    *   `get_events(game_id: str, agent_id: str)`: 为外部 AI Agent 提供按权限拉取历史事件的接口（内置可见性过滤）。优先查 Redis，不足时穿透查 DB。
    *   `_xread_events(...)`: 为外部 AI Agent 提供增量阻塞拉取新事件的接口。

### 2.3 Sequence Generator (序列号生成器)
基于 Redis 的 `INCR` 命令实现，保证多进程环境下的全局唯一性和严格单调递增。

## 3. 可见性路由策略 (Visibility Routing)

Event Bus 在分发事件或响应拉取请求时，必须执行以下过滤逻辑：

*   **PUBLIC (公开)**：分发给所有内部订阅者；任何 Agent 均可拉取。
*   **PRIVATE (私密)**：仅分发给相关的内部订阅者；仅允许 `target_agents` 列表中的 Agent 拉取。
*   **FACTION (阵营)**：仅分发给相关的内部订阅者；仅允许属于特定阵营的 Agent 拉取（需结合 GameState 判断）。

## 4. 存储与降级策略 (Storage & Degradation)

*   **热数据 (Redis Stream)**：作为高性能的消息队列和近期事件缓存，设置 `MAXLEN` 近似裁剪（如保留最近 1000 条）。
*   **冷数据 (PostgreSQL)**：作为 Event Sourcing 的最终事实来源，保存全量历史事件。
*   **降级机制**：当 Redis 不可用时，`publish` 仍会尝试将事件分发给内部订阅者（包括 DB 持久化），确保“事实不可丢”；`get_events` 查询时若 Redis 失败或数据不足，会自动回退（穿透）到 PostgreSQL 进行全量查询。
